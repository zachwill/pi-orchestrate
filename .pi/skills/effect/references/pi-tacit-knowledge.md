# Effect + Pi Tacit Knowledge

Use this note for Pi SDK, extension, session, and orchestration work. It records architectural evidence from public repositories, not a catalogue of APIs or a mandate to rewrite working direct TypeScript.

## Evidence discipline and version caveat

The examples span **Effect 3.20, 3.21, and v4 betas**; pi-orchestrate pins its own v4 beta. Those dependency labels describe manifests and lockfiles at the inspected revisions, not current branches or interchangeable APIs. Check the target lockfile and locally pinned Effect source before copying names or semantics for services, scopes, runtimes, fibers, and schemas.

Evidence was inspected at: `joelhooks/pi-effect@dde82d11fbc983c099a40862c3e8bd0936666f5e`, `mattpocock/sandcastle@e99f832f26dc9d245c019a9ddd19fa5dee792427`, `mmlngl/pi-bakery@375a27caae8cae1b7d6185e256491f2f8d8d9133`, `bobcats/pi-extensions@37a34d0d48294a439a5ef131f91214aec29fe9c6`, `drsh4dow/.pi@75a763710072a62cf7714775b353e1e7dd647613`, `gannonh/kata-code@351e4fa9d9049c36c7af56c76243624fa966f8c3`, `Derek-X-Wang/pihub@f005adaa6c74cdf768321a84eb2712d9b0ac30ab`, `jpenilla/BubbleBuddy@a42921b013320ac81139f2523a57ac4a2b330479`, `mattiacerutti/supernova@7bd70302e880016af599df3713db7bf6b519b995`, `getlarge/themoltnet@e65c479b36ad13728179ebc9d3e03dc7e31b93ec`, and `faizhasim/tele-kb-bot@4d56e7b8d06151b26fdf572dadc41182c8704d27`. Pinned links preserve that evidence; production code still shows constraints rather than a mandate, and experiments require independent validation.

## Tier 1: practices to apply now

- **Keep Effect internal and Promise/callback APIs public where Pi requires them.** Build the workflow as one Effect, run it at the Pi `execute`/hook/session boundary, and preserve the caller's `AbortSignal.reason`. Sandcastle's public `run()` performs internal Effect work and exits through `Effect.runPromise`, then calls `signal.throwIfAborted()` ([`src/run.ts`](https://github.com/mattpocock/sandcastle/blob/e99f832f26dc9d245c019a9ddd19fa5dee792427/src/run.ts)); drsh4dow's delegate does the same around a scoped child session ([`agent/extensions/delegate/runtime.ts`](https://github.com/drsh4dow/.pi/blob/75a763710072a62cf7714775b353e1e7dd647613/agent/extensions/delegate/runtime.ts)).
- **Make scopes match product lifetimes.** A scope should mean process host, owned session, retained worker, or one operation—not merely “some cleanup is needed.” BubbleBuddy creates an explicit scope for a retained Pi channel session and exposes `close` as `Scope.close` ([`packages/bubblebuddy/src/pi-session/session.ts`](https://github.com/jpenilla/BubbleBuddy/blob/a42921b013320ac81139f2523a57ac4a2b330479/packages/bubblebuddy/src/pi-session/session.ts)); its channel runtime closes that session from the channel-runtime finalizer ([`packages/bubblebuddy/src/channels/channel-runtime.ts`](https://github.com/jpenilla/BubbleBuddy/blob/a42921b013320ac81139f2523a57ac4a2b330479/packages/bubblebuddy/src/channels/channel-runtime.ts)).
- **Separate operation cancellation from reusable-resource disposal.** Aborting a prompt/turn is not closing its reusable session. BubbleBuddy has distinct `abort`, `close`, and channel discard paths in the two files above. drsh4dow interrupts `child.prompt`, invokes `child.abort`, unsubscribes, and only then releases with `child.dispose` ([`runtime.ts`](https://github.com/drsh4dow/.pi/blob/75a763710072a62cf7714775b353e1e7dd647613/agent/extensions/delegate/runtime.ts)). Keep this distinction visible in names, states, and tests.
- **Pass configured instances as values; reserve Context tags for shared capabilities.** A selected worker definition, model, provider instance, catalog snapshot, or per-request policy is data. Kata Code's factories return configured `SandboxProvider` values—see `makeDockerSandboxProvider` and `makeVercelSandboxProvider`—instead of minting a tag per configured provider ([Docker](https://github.com/gannonh/kata-code/blob/351e4fa9d9049c36c7af56c76243624fa966f8c3/packages/sandbox-docker/src/DockerSandboxProvider.ts), [Vercel](https://github.com/gannonh/kata-code/blob/351e4fa9d9049c36c7af56c76243624fa966f8c3/packages/sandbox-vercel/src/VercelSandboxProvider.ts)). Tags remain useful for stable capabilities such as stores and SDK adapters.
- **Normalize a shared raw event source once only when multiple independent consumers need it.** Pi-orchestrate's current direct, synchronous derivation is the right fit: worker observations update activity/usage/presentation, while domain settlements come from runtime state transitions and are not another projection of those observations. Do not collapse observations and settlements into one event type. If independent transcript, widget, usage, or diagnostics consumers later share the same raw source, one canonical representation can prevent divergent parsing. Supernova demonstrates that larger shape with a revisioned `SessionStreamEvent` union ([`watch-events.ts`](https://github.com/mattiacerutti/supernova/blob/7bd70302e880016af599df3713db7bf6b519b995/packages/contracts/src/session-runtime/procedures/watch-events.ts)), one PubSub ([`session-event-bus.ts`](https://github.com/mattiacerutti/supernova/blob/7bd70302e880016af599df3713db7bf6b519b995/packages/agent-runtime/src/layers/session-runtime/internal/session-event-bus.ts)), and live-turn/snapshot projections ([`pi-session-runtime.ts`](https://github.com/mattiacerutti/supernova/blob/7bd70302e880016af599df3713db7bf6b519b995/packages/agent-runtime/src/layers/session-runtime/internal/pi-session-runtime.ts)); its Effect v4 APIs are beta-era.
- **Treat retained sequential workers as an existing product scope.** Pi-orchestrate already retains one reusable worker identity/session across sequential generations, with `worker_abort` scoped to active work and `worker_close` scoped to the retained resource. Preserve that explicit lifetime even without introducing an Effect `Scope`; aggregate more resources under a scope only if teardown complexity makes it useful.
- **Give discovery and probes reduced authority.** A catalog/resource probe should not accidentally load extensions, prompts, themes, or broad tools. Supernova's discovery loader explicitly disables those surfaces and re-enables only selected skill paths ([`packages/agent-runtime/src/layers/pi-config.ts`](https://github.com/mattiacerutti/supernova/blob/7bd70302e880016af599df3713db7bf6b519b995/packages/agent-runtime/src/layers/pi-config.ts)). Kata Code's provider `testConnection`/health probes use cheaper, disposable paths distinct from durable instances ([Vercel provider](https://github.com/gannonh/kata-code/blob/351e4fa9d9049c36c7af56c76243624fa966f8c3/packages/sandbox-vercel/src/VercelSandboxProvider.ts), [`sdk.ts`](https://github.com/gannonh/kata-code/blob/351e4fa9d9049c36c7af56c76243624fa966f8c3/packages/sandbox-vercel/src/sdk.ts)).
- **Enforce source-first rules on every agent-facing surface.** joelhooks' pi-effect repeats the rule in the skill, tool description/guidelines, and `before_agent_start` prompt injection ([`skills/pi-effect/SKILL.md`](https://github.com/joelhooks/pi-effect/blob/dde82d11fbc983c099a40862c3e8bd0936666f5e/skills/pi-effect/SKILL.md), [`extensions/pi-effect.ts`](https://github.com/joelhooks/pi-effect/blob/dde82d11fbc983c099a40862c3e8bd0936666f5e/extensions/pi-effect.ts)); the bounded detector/hydrator/search implementation is in [`extensions/effect-source-workspace.ts`](https://github.com/joelhooks/pi-effect/blob/dde82d11fbc983c099a40862c3e8bd0936666f5e/extensions/effect-source-workspace.ts). A rule present only in prose is easier for an agent to miss.
- **Minimize capabilities, not only instructions.** Tele KB Bot disables built-ins and filesystem discovery, then installs only compiled-in memory tools ([`src/pi/session-factory.ts`](https://github.com/faizhasim/tele-kb-bot/blob/4d56e7b8d06151b26fdf572dadc41182c8704d27/src/pi/session-factory.ts), [`src/pi/extensions.ts`](https://github.com/faizhasim/tele-kb-bot/blob/4d56e7b8d06151b26fdf572dadc41182c8704d27/src/pi/extensions.ts)); drsh4dow removes recursive delegation tools before prompting a child ([`agent/extensions/delegate/runtime.ts`](https://github.com/drsh4dow/.pi/blob/75a763710072a62cf7714775b353e1e7dd647613/agent/extensions/delegate/runtime.ts)). The observed tool-loop designs support capability minimization: a prompt saying “do not use X” is weaker than making X unavailable.

## Pi-orchestrate ownership matrix

Use this matrix when deciding where a scope or finalizer belongs. It describes ownership, not a required Effect refactor.

| Thing | Owner / product lifetime | Cancellation vs. close | Expected boundary |
|---|---|---|---|
| Process host | Pi process; survives extension reload attachment changes | Process shutdown closes host-held resources best-effort | Process-scoped host value; Promise lifecycle adapters |
| Scheduler | One process-host/runtime-scoped scheduler shared across owner attachments; never one scheduler per owner attachment | Parent-turn cancellation affects that owner's inline operations; runtime shutdown settles admitted work across attachments | Process-scoped runtime boundary; internal Effect only where interruption/finalization adds value |
| Worker workflow | One admitted generation/wave result | Abort interrupts the active generation; settlement completes it | Scoped operation, surfaced as Promise to the tool |
| Retained session | Reusable worker identity across generations | `worker_abort` stops active work; `worker_close` disposes the ready session | Explicit retained scope/close handle owned by runtime |
| Prompt generation | One child creation or generation | Cancellation abandons generation; no independent durable resource | Plain value/effect computed under worker workflow |
| Catalog snapshot | One atomic preflight/admission decision | Immutable after admission; refresh creates a new snapshot | Validated value passed to runtime, not a Context tag |

## Tier 2: conditional patterns for future needs

- **Aggregate retained resources under an Effect scope only when teardown grows more complex.** The retained sequential-worker lifetime already exists in pi-orchestrate. BubbleBuddy's `createPiChannelSession` shows how to implement that lifetime when one worker must close subscriptions, resource loaders, MCP clients, and a Pi session together. Do not add an Effect scope merely to wrap one disposable.
- **Use `acquireRelease`/`acquireUseRelease` for nested resources with real teardown order.** Sandcastle intentionally nests worktree and sandbox acquisition so the inner handle closes before outer worktree cleanup ([`src/SandboxFactory.ts`](https://github.com/mattpocock/sandcastle/blob/e99f832f26dc9d245c019a9ddd19fa5dee792427/src/SandboxFactory.ts)). bobcats scopes a temp prompt file and child process cancellation ([`subagent/temp-effect.ts`](https://github.com/bobcats/pi-extensions/blob/37a34d0d48294a439a5ef131f91214aec29fe9c6/subagent/temp-effect.ts), [`subagent/process-effect.ts`](https://github.com/bobcats/pi-extensions/blob/37a34d0d48294a439a5ef131f91214aec29fe9c6/subagent/process-effect.ts)). Apply when partial acquisition and ordering are observable.
- **Add queues/PubSub only when consumers or backpressure demand them.** Supernova's event bus is justified by RPC streaming and multiple projections. BubbleBuddy's scoped `PriorityDrainableWorker` uses transactional queues and `forkScoped` for prioritized draining ([`packages/bubblebuddy/src/shared/priority-drainable-worker.ts`](https://github.com/jpenilla/BubbleBuddy/blob/a42921b013320ac81139f2523a57ac4a2b330479/packages/bubblebuddy/src/shared/priority-drainable-worker.ts)). A synchronous state listener does not need this machinery.
- **Build one application layer at a true executable boundary.** Pihub wires a CLI-wide dependency graph and runs it once with `BunRuntime.runMain` ([`apps/cli/src/cli.ts`](https://github.com/Derek-X-Wang/pihub/blob/f005adaa6c74cdf768321a84eb2712d9b0ac30ab/apps/cli/src/cli.ts)); its service modules expose narrow test layers, for example [`packages/core/src/paths.ts`](https://github.com/Derek-X-Wang/pihub/blob/f005adaa6c74cdf768321a84eb2712d9b0ac30ab/packages/core/src/paths.ts). This fits a CLI better than repeatedly constructing runtimes inside Pi callbacks.
- **Use a process-level fiber owner only if detached work truly has process lifetime.** bobcats' [`subagent/async-owner.ts`](https://github.com/bobcats/pi-extensions/blob/37a34d0d48294a439a5ef131f91214aec29fe9c6/subagent/async-owner.ts) tracks detached fibers and interrupts them on shutdown. For pi-orchestrate, owner-scoped delivery and retained sessions still need explicit records; a Fiber map alone is not the domain model.
- **Use Effect CLI in Pi commands only if parsing complexity earns it.** pi-bakery's [`packages/effect-playground/extensions/effect-playground.ts`](https://github.com/mmlngl/pi-bakery/blob/375a27caae8cae1b7d6185e256491f2f8d8d9133/packages/effect-playground/extensions/effect-playground.ts) adapts a Pi command to Effect CLI and injects the command context with [`ctx.ts`](https://github.com/mmlngl/pi-bakery/blob/375a27caae8cae1b7d6185e256491f2f8d8d9133/packages/effect-playground/extensions/ctx.ts). It is explicitly a playground/experiment, not production guidance.

## Tier 3: anti-patterns and cautions

- **Do not infer disposal from removal.** Tele KB Bot's idle/LRU eviction deletes registry entries without disposing their sessions, while explicit `remove` does dispose ([`src/daemon/session-registry.ts`](https://github.com/faizhasim/tele-kb-bot/blob/4d56e7b8d06151b26fdf572dadc41182c8704d27/src/daemon/session-registry.ts)). Treat this as a caution, not a pattern: every eviction path must execute the same close policy.
- **Do not conflate interruption cleanup with durable shutdown.** An `Effect.async` canceler can kill a running subprocess, as in bobcats' `process-effect.ts`; it cannot by itself express whether a reusable worker should remain ready after a generation.
- **Do not swallow every finalizer failure by habit.** Best-effort cleanup can be correct, but teardown order and failure policy deserve characterization tests: startup fails after subscription, abort fails, unsubscribe throws, dispose throws, and shutdown repeats. Sandcastle encodes dirty-worktree preservation versus handle close in nested releases; drsh4dow's delegate makes abort/unsubscribe/dispose separate releases. Preserve these distinctions.
- **Do not promote experiments into house style.** pi-bakery labels its Effect extension a playground. joelhooks/pi-effect is source-workflow tooling, not an Effect-based orchestration implementation. bobcats/pi-extensions is valuable migration/test evidence but remains a collection of extensions rather than one stable application architecture.
- **Do not cite themoltnet as Effect evidence: it does not actually use Effect.** [`getlarge/themoltnet`](https://github.com/getlarge/themoltnet/tree/e65c479b36ad13728179ebc9d3e03dc7e31b93ec) has no `effect` dependency/import in the investigated source. Its relevant lesson is cross-surface guidance distribution—see [`.github/workflows/skill-sync.yml`](https://github.com/getlarge/themoltnet/blob/e65c479b36ad13728179ebc9d3e03dc7e31b93ec/.github/workflows/skill-sync.yml), [`.github/scripts/check-skill-sync.sh`](https://github.com/getlarge/themoltnet/blob/e65c479b36ad13728179ebc9d3e03dc7e31b93ec/.github/scripts/check-skill-sync.sh), and [`.pi/model-guidance.md`](https://github.com/getlarge/themoltnet/blob/e65c479b36ad13728179ebc9d3e03dc7e31b93ec/.pi/model-guidance.md)—not Effect runtime design.
